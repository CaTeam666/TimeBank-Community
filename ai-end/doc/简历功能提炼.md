# 时间银行互助养老平台 - 简历功能提炼备忘录

## 📌 项目概述
- **项目名称**：时间银行互助养老平台（后端）
- **核心技术栈**：Spring Boot 3.1.5、MyBatis-Plus 3.5.5、MySQL、Redis、JDK 17
- **项目定位**：聚焦于智慧养老体系的“时间互助”模式，打造包含需求发布、志愿接单、亲属代办与积分流转的分布式业务级服务接口。

---

## 🚀 可以写进简历的核心亮点与功能模块（8大核心）

### 1. 复杂状态机管理与全链路业务闭环（亮点：业务架构能力）
- **功能描述**：设计并实现了“发布服务 -> 志愿接单 -> 提交存证 -> 服务验收 -> 纠纷仲裁”的核心任务流转系统。
- **简历话术提炼**：
  > “主导设计养老互助业务的**核心任务流转引擎**，引入状态机管理模型处理任务与亲属关系的十余种状态变更；实现了从前端需求触发到虚拟资产最终清算的完整业务闭环，并配套实现了**历史服务存证**与**管理员长效仲裁介入**机制，确保平台订单具备极高容错性与可追溯性。”

### 2. 双端 JWT 鉴权与个性化权限穿透设计（亮点：鉴权与安全架构）
- **功能描述**：实现管理后台（Admin）与移动端（Client）的两套独立认证体系，并突破性支持子女无缝代理操作。
- **简历话术提炼**：
  > “基于 Spring 拦截器（Interceptor）和 Redis 构建了**双端隔离的分布式 JWT 认证系统**，针对不同端实现了生命周期的精准控制。针对适老化特殊场景，创新性设计了跨层级的**权限穿透**逻辑（基于 `X-Real-User-Id` 动态解析），实现了‘子女静默代理老人’的安全操作层切换，大幅提升适老化系统易用性。”

### 3. 高性能排行榜与时间币资产引擎（亮点：中间件与高并发场景应用）
- **功能描述**：基于 Redis 构建实时排行榜，并实现了类似钱包体系的虚拟货币“时间币”的流转账单系统。
- **简历话术提炼**：
  > - **性能优化**：充分利用 Redis 数据结构特性（ZSet、Hash），负责主导并攻克了高并发下的**志愿者接单实时排行榜**（5分钟级级刷新）及关键业务指标的高效统计，成功将海量读压力从关系型数据库层剥离。
  > - **资产安全引擎**：从零构建了基于“时间币”的虚拟资产流转体系（涵盖发布冻结、完成发放、系统调节、商城核销消费等 4 层强关联流水模型），结合数据库底层事务约束保证了资金转移的强一致性与随时可对账审核能力。

### 4. 高可用定时任务调度器与补偿机制（亮点：系统可靠性保障）
- **功能描述**：独立使用系统级 Scheduler 维护底层数据的时效性，定期流转失效任务并结算数据。
- **简历话术提炼**：

### 5. 工程化规范与统一接口层封装标准（亮点：代码素养与开发规范）

## 🕵️ 项目潜在“漏洞”分析与应对方案（用于反杀面试官）

这是一个非常典型的高并发单体架构原型项目。如果求职面试时提到涉及了资金相关（时间币流转）与定时任务模块，技术官极有可能循循善诱、深挖并发与分布式架构的痛点缺陷（俗称压力测试/挖坑）。**请务必牢记以下体系漏洞及其架构演进的整改反思与优化思路，它将成为您“反客为主”争取更高薪资与架构师级评价的重要谈资：**

### 漏洞一：Spring Scheduler 定时任务单点与并发风暴
- **业务现状**：系统中共包含 5 个 `@Scheduled` 高频定时应用（如每月1号的排名奖金瓜分、每隔5分钟重新梳理榜单等）。
- **缺陷剖析**：单机跑没事，但当项目为了容灾部署为两节点或以上的服务器构建集群高可用架构（HA）模式时。一旦时钟重合，所有实例都会在同时抢跑任务——不仅无谓地使数据库 IO 耗费翻番，更是会导致可怕的**“分两次钱”、“资金超发”、“重复写入榜单记录”的业务雪崩事故！**
- **应对演练 & 技术反思设计**：“我当前的起步版本用 `@Scheduled` 是为了极速交付和验证闭环模型；但经过我的压测和全盘梳理并发现**这是横向扩充的核心瓶颈**。如果接下来要在该架构下拓展，我的架构重整核心思路是直接植入针对各类任务 `Key` 的 **Redisson 也就是 Redis 提供的分布式定时任务排他锁机制**，先占锁再处理数据；甚至为业务长期考虑的话将调度剥离出单体服务直接迁移给 `XXL-JOB` 这类专业的分布式监控与任务流转处理分发中心。”

### 漏洞二：大事务阻塞以及“缺少全站 MQ 异步解耦”
- **业务现状**：目前用户端的“最终完结任务”或者结算节点，会使我们的同步业务执行一堆相关写盘请求：刷新订单->改写排行榜->调整两边用户的钱包表->插一笔流水账单->在系统库中强插多封站内信通知....等操作统统囊括并压附于同一个巨型的 `@Transactional` 事务代码段内。
- **缺陷剖析**：这绝对是架构禁忌里的**“死亡型长事务阻塞”**。如果写通知或者计算积分的那一端代码由于网络颠簸发生微弱卡滞，整整一条链式的庞大更新业务就会卡顿、事务一直未能提交，同时在后端一直耗尽数据库的关键资源与连接锁，一旦峰值并发洪峰袭来甚至直接导致 `DruidDB` 连接池全面告急而整个平台全面宕机失去响应。
- **应对演练 & 技术反思设计**：“在通过基准测试发现这根主数据传输链路拖泥带水的弱点后，我其实制定了详细的主程解绑计划，决定 **彻底拥抱 RocketMQ 或 RabbitMQ 等 MQ消息中间件，作长线防波堤削峰填谷和彻底的弱关联解耦**。我需要核心接口保证极致响应：主转账链路一旦更新最终主订单和锁定钱包即可强提交，然后发出一个包含核心变动的事件全频道 MQ 通知广播；其他像查重算分、积分日志登记记录以及繁杂的通知信统统被流转下发去排队。经这套操作我能把原来拖到 300 毫秒的危险主干响应提优压缩突破 50ms 大关！”

### 漏洞三：积分商城资产体系的防抖保护与“并发超扣/超卖”防穿透
- **业务现状**：时间币核销、以及退换核对在目前直查直写的单一验证链路。
- **缺陷剖析**：如果前端按钮连点且弱网络环境下，亦或是恶意灰产用户脚本截获数据包，高频将报文朝同一个接口单秒倾泻 20 次兑换相同爆款礼物的请求。如果是“判断足够后立刻执行余额划减”这样的平凡思维编码，就会由于锁没获取到而在极小的时间窗里让单个兑换项瞬间穿透核准层，直接导致库存被打为大缺口负数，或时间币账号恶意刷大亏空。
- **应对演练 & 技术反思设计**：“涉及底层钱包与库存扣减场景（特别是更新 `tb_product` 和 `tb_coin_log` 等涉及资产表），我在系统建设最内核对数据库事务增补并落网了最绝对底层的**乐观锁层级防护拦截逻辑（MyBatis-Plus 的 `@Version` 并发控制）**，即使代码再如何乱写我在最底层必须配合安全的加带安全边界 SQL（强挂条件：`Update stock=stock-1 where id=? and stock>0`）。再此基础上如果平台流量继续翻升，我会通过**利用 Redis 对于 userId 与操作动作实现前置性的防连点拦截（去重），辅加上 Lua 高响应保证库存清减的防并发超卖锁隔离**。”

### 漏洞四：单节点 Redis 实时排行的严重“大 Key”热载风险预演
- **业务现状**：平台一整个大月的实时高频交互核心 ZSet 数据模型被粗犷地集中塞入了单一缓存 Redis ZSet 映射下，名为 `ranking:realtime:{yyyy-MM}` 对象里。
- **缺陷剖析**：在应用测试时问题不显。假定如果运营大推让平台单月真实活跃人口疯涨几万量级以上，这单单一条月榜所在的单个 ZSet 项将在 Redis 库里急骤演变成致命级别的超巨型隐患**“大 Key”**，由于 Redis 命令的单线程吞吐法则，一旦对这条超大内容块执行重计算或者批量调阅操作时会阻断所有连接当前 Redis 的别人查取请求并全线丢包；再更严重的，假设没进行严格地持久化挂载，只要一次单点内存奔溃服务器重启就会遭受毁灭打击被完全抹除记忆并且不可复原。
- **应对演练 & 技术反思设计**：“既然是高并发排行榜我就绝不会放任这种雪球式滚雪风险持续膨胀！如果一旦侦测到容量阀门预警，我的第一个应急重构策略就是“实施严管截断筛选与按分化区切割热区（打散Key分流）”，最直接的做法是例如积分必须先上 200 的有效玩家才会进 Redis ，并且全省级别不能混存而是根据城市区按地分榜打散；另一方面我必须结合“双冗灾快照降级”，确保有每日凌晨冷任务异步将核心排位定期刷刷固化进数据库集群或者 Elasticsearch 等冷储存中心。确保哪管 Redis 服务器机房拔了电依然有最后的安全逃生兜底容错渠道恢复整个局势！”
