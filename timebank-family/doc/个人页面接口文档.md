# 个人页面接口文档

## 1. 获取我的接单列表

**接口描述**: 获取当前用户作为志愿者接取的任务列表

**请求路径**: `/api/task/my/accepted`

| data.coins | number | 悬赏积分 |

| data.evidencePhotos | array | 服务凭证图片URL列表 |
| data.checkInTime | string | 签到时间 (ISO格式) |

| data.volunteerId | string | 志愿者ID |

**响应示例**:
```json
{
  "code": 200,
  "message": "查询成功",
  "data": {
    "taskId": "1735467890456",
    "title": "帮我买袋米",
    "description": "需要5kg大米",
    "coins": 20,
    "status": 2,
    "evidencePhotos": [
      "http://example.com/evidence1.jpg",
      "http://example.com/evidence2.jpg"
    ],
    "checkInTime": "2025-01-02T09:30:00.000Z",
    "finishTime": "2025-01-02T10:15:00.000Z",

    "volunteerName": "李志愿",
    "volunteerAvatar": "http://example.com/avatar.jpg"
  }
}
```

**数据库操作**:
- 关联查询 `tb_task`、`tb_task_evidence`、`sys_user` 表
- SQL示例:
```sql
SELECT 
  t.id, t.title, t.description, t.price, t.status, t.volunteer_id,
  u.nickname as volunteer_name, u.avatar as volunteer_avatar,
  e.image_url, e.check_in_time, e.create_time as finish_time
FROM tb_task t
LEFT JOIN sys_user u ON t.volunteer_id = u.id
LEFT JOIN tb_task_evidence e ON t.id = e.task_id AND e.is_deleted = 0
WHERE t.id = #{taskId} AND t.is_deleted = 0
```

## 7. 确认验收

**接口描述**: 发布者确认验收任务，将任务状态变更为"已完成"，并完成积分结算

**请求方法**: `POST`
**请求路径**: `/api/task/review/confirm`

**请求参数 (JSON)**:

| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| taskId | string | 是 | 任务ID | 1735467890456 |
| userId | string | 是 | 用户ID（发布者） | 1001 |
| rating | number | 是 | 评分 (1-5) | 5 |
| review | string | 否 | 评价内容 | 服务态度很好 |

**响应参数 (JSON)**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| code | number | 状态码 (200成功) |
| message | string | 响应信息 |
| data | boolean | 是否验收成功 |

**响应示例**:
```json
{
  "code": 200,
  "message": "验收成功",
  "data": true
}
```

**数据库操作**:
1. 更新 `tb_task` 表的 `status` 字段为 `3`（已完成）
2. 扣除发布者积分：更新 `sys_user.balance` (发布者)
3. 增加志愿者积分：更新 `sys_user.balance` (志愿者)
4. 记录发布者支出流水：向 `tb_coin_log` 插入一条记录 (type=2, amount=-coins)
5. 记录志愿者收入流水：向 `tb_coin_log` 插入一条记录 (type=1, amount=+coins)

**业务逻辑**:
- 任务状态必须为 `2`（待确认）才能验收
- 操作用户必须是任务发布者
- 需在事务中完成所有操作，确保数据一致性

## 8. 提交志愿者评价

**接口描述**: 发布者对完成任务的志愿者提交评价

**请求方法**: `POST`
**请求路径**: `/api/task/review/submit`

**请求参数 (JSON)**:

| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| taskId | string | 是 | 任务ID | 1735467890456 |
| publisherId | string | 是 | 发布者ID（评价人） | 1001 |
| volunteerId | string | 是 | 志愿者ID（被评价人） | 2001 |
| rating | number | 是 | 评分 (1-5) | 5 |
| content | string | 否 | 评价内容 | 服务态度很好，非常满意 |

**响应参数 (JSON)**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| code | number | 状态码 (200成功) |
| message | string | 响应信息 |
| data | boolean | 是否提交成功 |

**响应示例**:
```json
{
  "code": 200,
  "message": "评价成功",
  "data": true
}
```

**数据库操作**:
1. 向 `tb_task_review` 表插入一条评价记录
2. 可选：更新志愿者的平均评分（如需统计）

**业务逻辑**:
- 任务状态必须为 `3`（已完成）才能评价
- 操作用户必须是任务发布者
- 每个任务只能评价一次（task_id 唯一约束）
- 评分范围必须在 1-5 之间

**SQL示例**:
```sql
INSERT INTO tb_task_review (task_id, publisher_id, volunteer_id, rating, content)
VALUES (#{taskId}, #{publisherId}, #{volunteerId}, #{rating}, #{content});
```
