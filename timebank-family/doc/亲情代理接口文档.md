# 亲情代理接口文档

## 接口概述

本文档定义亲情账号绑定及代理操作的相关接口，包括申请绑定、查询绑定列表、解绑、代理模式切换等功能。

---

## 1. 申请绑定亲情账号

**接口路径**: `POST /api/family/bind`

**功能说明**: 子女用户通过手机号申请绑定长者账号

**请求头**:
```
Authorization: Bearer {token}
Content-Type: application/json
```

**请求参数**:
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| phone | string | 是 | 被绑定长者的手机号 |
| relation | string | 是 | 关系类型(如:父亲、母亲、爷爷、奶奶) |
| proofImg | string | 否 | 证明材料图片URL |

**请求示例**:
```json
{
  "phone": "13800138000",
  "relation": "父亲",
  "proofImg": "https://example.com/proof.jpg"
}
```

**响应参数**:
| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | integer | 状态码 (200:成功) |
| message | string | 响应消息 |
| data | object | 响应数据 |
| data.relationId | integer | 关系记录ID |

**响应示例**:
```json
{
  "code": 200,
  "message": "绑定申请已提交，等待审核",
  "data": {
    "relationId": 123
  }
}
```

**错误码**:
| 错误码 | 说明 |
|--------|------|
| 400 | 参数错误 |
| 404 | 手机号对应的用户不存在 |
| 409 | 已存在绑定关系 |
| 500 | 服务器内部错误 |

## 4. 切换代理模式

**接口路径**: `POST /api/family/proxy/toggle`

**功能说明**: 开启或关闭代理模式，子女可代替长者进行操作

**请求头**:
```
Authorization: Bearer {token}
Content-Type: application/json
```

**请求参数**:
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| parentId | integer | 否 | 长者用户ID(为null时退出代理模式) |
| enable | boolean | 是 | true:开启代理 false:关闭代理 |

**请求示例**:
```json
{
  "parentId": 1001,
  "enable": true
}
```

**响应参数**:
| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | integer | 状态码 |
| message | string | 响应消息 |
| data | object | 响应数据 |
| data.proxyToken | string | 代理令牌(后续请求使用此token代表长者身份) |

**响应示例**:
```json
{
  "code": 200,
  "message": "已切换为代理模式",
  "data": {
    "proxyToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}
```

**错误码**:
| 错误码 | 说明 |
|--------|------|
| 400 | 参数错误 |
| 403 | 无绑定关系，无权代理 |
| 404 | 用户不存在 |
| 500 | 服务器内部错误 |

**ProxyToken结构说明**:

`proxyToken` 是一个 JWT Token，其 payload 包含以下字段：

```json
{
  "userId": 1001,           // 被代理人ID（老人ID）- 执行操作时使用此ID
  "realUserId": 2001,       // 实际操作人ID（子女ID）- 用于审计日志
  "isProxy": true,          // 代理标识
  "exp": 1234567890         // 过期时间（建议2小时）
}
```

**使用方式**:
- 前端获取 `proxyToken` 后存储到 `localStorage` 中
- 在代理模式下的所有API请求中，使用 `proxyToken` 替换普通 `token`
- 请求头格式：`Authorization: Bearer {proxyToken}`
- 后端解析 JWT 时，使用 `userId` 字段作为当前操作用户

## 4-1. 代理模式权限说明

**功能概述**: 代理模式下，子女可以代替长者进行**受限的**操作，以帮助不熟悉手机操作的老人使用系统。

**权限白名单**: 

子女在代理模式下**仅可以**访问以下接口：

### 任务相关
| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/api/task/publish` | POST | 发布任务 |
| `/api/task/my-published` | GET | 查看我发布的任务 |
| `/api/task/my-accepted` | GET | 查看我接的任务 |

### 用户相关
| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/api/user/balance` | GET | 查询积分余额 |

### 商城相关
| 接口路径 | 方法 | 说明 |
|---------|------|------|
| `/api/mall/products` | GET | 查看商品列表 |
| `/api/mall/exchange` | POST | 兑换商品 |
| `/api/mall/orders` | GET | 查看兑换订单 |

**禁止访问的接口**:

以下操作**不允许**在代理模式下执行，后端应返回 `403 Forbidden`：
- 修改用户个人信息（头像、昵称、手机号、密码等）
- 实名认证
- 解绑亲情关系
- 接取任务（以老人身份做志愿者）
- 申诉、评价等其他敏感操作

**后端权限验证逻辑**:
```
1. 解析请求中的 Authorization Token
2. 如果 token 中包含 isProxy=true，进行权限白名单检查
3. 如果请求的接口在白名单中，允许访问，使用 userId 作为操作人
4. 如果请求的接口不在白名单中，返回 403 错误
5. 记录 realUserId 到审计日志，便于追溯实际操作人
```

## 6. 审核绑定申请

**接口路径**: `POST /api/family/review`

**功能说明**: 被绑定方审核绑定申请(通过或拒绝)

**请求头**:
```
Authorization: Bearer {token}
Content-Type: application/json
```

**请求参数**:
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| relationId | integer | 是 | 关系记录ID |
| action | string | 是 | 审核动作: "approve"(通过) 或 "reject"(拒绝) |
| rejectReason | string | 否 | 拒绝原因(action为reject时必填) |

**请求示例**:
```json
{
  "relationId": 125,
  "action": "approve"
}
```

或

```json
{
  "relationId": 126,
  "action": "reject",
  "rejectReason": "身份信息不符"
}
```

**响应示例**:
```json
{
  "code": 200,
  "message": "审核成功"
}
```

**错误码**:
| 错误码 | 说明 |
|--------|------|
| 400 | 参数错误 |
| 403 | 无权限操作 |
| 404 | 申请记录不存在 |
| 500 | 服务器内部错误 |

## 数据库操作

### 涉及的表

1. **sys_user_relation** (系统用户关系表)
   - 查询操作: 获取列表、获取待审核申请、验证绑定关系
   - 插入操作: 创建绑定申请
   - 更新操作: 审核绑定申请、解绑(逻辑删除)

2. **sys_user** (系统用户表)
   - 查询操作: 获取用户信息、验证用户存在性

### 关键SQL说明

**获取亲情账号列表**:
```sql
SELECT 
  u.id, u.nickname, u.avatar, u.balance, u.phone,
  r.id as relationId, r.relation, r.status
FROM sys_user_relation r
INNER JOIN sys_user u ON r.parent_id = u.id
WHERE r.child_id = ? 
  AND r.status = 1 
  AND r.is_deleted = 0
  AND u.deleted = 0
```

**获取待审核申请**:
```sql
SELECT 
  r.id as relationId, r.relation, r.proof_img as proofImg, r.create_time as createTime,
  u.id as childId, u.realName as childName, u.avatar as childAvatar, u.phone as childPhone
FROM sys_user_relation r
INNER JOIN sys_user u ON r.child_id = u.id
WHERE r.parent_id = ? 
  AND r.status = 0 
  AND r.is_deleted = 0
ORDER BY r.create_time DESC
```

## 注意事项

1. **权限验证**: 所有接口都需要验证用户登录状态(token)
2. **手机号脱敏**: 返回手机号时应进行脱敏处理(如: 138****8000)
3. **状态流转**: 绑定申请状态流程为: 
   - 待管理员审核(0) → 待老人确认(1) → 已绑定(2)
   - 或 待管理员审核(0) → 已拒绝(3)
   - 或 待老人确认(1) → 已拒绝(3)
4. **代理令牌**: 
   - 代理令牌(proxyToken)应包含被代理人ID(userId)和实际操作人ID(realUserId)
   - ProxyToken应设置合理的过期时间（建议2小时）
   - 前端应在代理模式下使用 proxyToken 替换普通 token
5. **代理权限控制**: 
   - **严格执行白名单机制**，代理模式仅可访问指定的接口
   - 后端必须验证 proxyToken 中的 `isProxy` 标识
   - 非白名单接口收到 proxyToken 时应返回 `403 Forbidden`
   - 记录 realUserId 到审计日志，便于追溯实际操作人
6. **逻辑删除**: 解绑操作应使用逻辑删除(is_deleted=1)，保留历史记录
7. **消息通知**: 当有待确认绑定申请(status=1)时，个人页面应显示红点提示
8. **用户体验**: 
   - 前端应在所有关键页面显示明显的代理提示，避免用户混淆
   - 代理模式下的操作成功提示应体现代理关系（如："已为 XXX 发布任务"）
